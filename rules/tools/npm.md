# npm Conventions

## Dependencies — MANDATORY

**Never add, remove, or change dependencies (including version changes) without asking first.** If you think a dependency change is needed, explain why and wait for explicit approval.

Exception: once the user explicitly asks you to add, remove, or update a specific dependency in this conversation, you may do so without asking again for that same change.

This applies to both `dependencies` and `devDependencies`.

## package-lock.json

- Always commit `package-lock.json` — it locks the exact dependency tree
- Never manually edit `package-lock.json` — it is generated by npm
- Use `npm ci` in CI/CD and scripts — installs from the lockfile exactly, fails if out of sync
- Use `npm install` only in local dev when intentionally updating the lockfile

## dependencies vs devDependencies

- `dependencies` — required at runtime (frameworks, libraries the app ships with)
- `devDependencies` — build/test/lint tools only, never needed at runtime

## scripts

- Keep names consistent: `build`, `test`, `start`, `dev`, `lint`
- Chain commands with `&&`, not `;` — failure should stop the chain

## Security

- Never run `npm audit fix --force` without asking — it can silently break APIs
- Review `npm audit` output and ask before acting on it
